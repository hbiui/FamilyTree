## 🔬 验证清单

### 验证一：异常捕获和 Toast 提示

#### 目标

验证不同类型的网络异常是否被正确捕获并显示对应的 Toast 提示。

#### 验证步骤

1. **进入设置页面**
   - 打开「设置」标签
   - 找到「网络模拟调试」面板

2. **启用网络模拟**
   - 点击顶部开关启用网络模拟

3. **模拟 403 禁止访问**
   - 选择「403禁止」模式
   - 设置失败率为 100%
   - 进入「成员」页面或任何发起 API 请求的页面
   - ✅ **验证结果**：应该看到红色 Toast 提示「权限不足，无法执行此操作」

4. **模拟 404 找不到**
   - 回到设置页面，选择「404找不到」模式
   - 设置失败率为 100%
   - 尝试访问一个不存在的成员详情
   - ✅ **验证结果**：应该看到 Toast 提示「请求的资源不存在」

5. **模拟 500 错误**
   - 选择「500错误」模式
   - 设置失败率为 100%
   - 发起 API 请求
   - ✅ **验证结果**：应该看到 Toast 提示「服务器暂时不可用，请稍后重试」，并且会自动重试

6. **模拟离线**
   - 选择「离线」模式
   - 发起 API 请求
   - ✅ **验证结果**：应该看到红色 Toast 提示「网络不稳定，请检查您的网络连接」，同时顶部显示「离线模式」横幅

#### 预期结果

| 模拟模式 | 预期 Toast 消息 | 自动重试 |
|----------|----------------|----------|
| 离线 | 网络不稳定，请检查您的网络连接 | ✅ |
| 超时 | 请求超时，请重试 | ✅ |
| 500错误 | 服务器暂时不可用，请稍后重试 | ✅ |
| 403禁止 | 权限不足，无法执行此操作 | ❌ |
| 404找不到 | 请求的资源不存在 | ❌ |

### 验证二：自动重试和手动重试

#### 目标

验证关键操作失败后是否自动重试，以及用户是否可以手动重试。

#### 验证步骤

1. **配置网络模拟**
   - 在设置页面启用网络模拟
   - 选择「500错误」模式
   - 设置失败率为 70%（这样有时会成功，有时会失败）
   - 设置延迟为 2000ms

2. **发起关键操作**
   - 进行一个模拟的「保存成员」操作
   - 观察控制台日志（如果是 web 模式打开 DevTools）
   - ✅ **验证结果**：
     - 第一次失败后，应该显示「网络不稳定，已自动重试」或类似消息
     - 应该有指数退避延迟（每次重试间隔更长）
     - 最多重试 3 次

3. **手动重试**
   - 如果 3 次都失败，等待最终错误提示
   - ✅ **验证结果**：Toast 上应该有「重试」按钮
   - 点击「重试」按钮
   - ✅ **验证结果**：操作应该重新发起

4. **网络恢复后自动同步（如果与离线功能结合）**
   - 先在离线模式下进行几个操作
   - 然后恢复在线
   - ✅ **验证结果**：顶部横幅会显示待同步数量，并且可以点击「立即同步」

---

### 验证三：防重复点击

#### 目标

验证在按钮被点击后，直到操作完成前，是否能防止重复点击。

#### 验证步骤

1. **准备测试环境**
   - 确保网络模拟已启用
   - 选择「慢速网络」模式
   - 设置延迟为 3000ms（足够长以便测试重复点击）

2. **测试防重复点击**
   - 找到一个有「保存」或「提交」操作的按钮
   - 快速点击按钮多次（比如在 1 秒内连续点击 5 次）
   - ✅ **验证结果**：
     - 只有第一次点击会被处理
     - 按钮可能显示加载状态（如果实现了）
     - 其他点击被忽略
   - 等待 3 秒操作完成
   - ✅ **验证结果**：只有一个 API 请求被发送

3. **验证操作完成后恢复**
   - 操作成功完成后
   - 再次点击按钮
   - ✅ **验证结果**：按钮应该可以正常点击，发起新的操作

### 验证四：网络异常模拟（Charles/Flipper 方案）

#### 使用 Charles Proxy 进行网络拦截

如果需要更真实的网络测试（不仅仅是模拟），可以使用 Charles Proxy：

1. **安装 Charles Proxy**
   - 下载地址：https://www.charlesproxy.com/

2. **配置代理**
   - 在手机或模拟器上配置 HTTP 代理指向 Charles
   - 安装 SSL 证书以支持 HTTPS 拦截

3. **模拟网络错误**
   - 打开 Charles，找到想要拦截的请求
   - 右键点击请求 → Breakpoints
   - 或者使用 Tools → Throttle Settings 模拟慢速网络
   - 或者在请求到达时手动设置响应状态码

#### 使用 Flipper 进行调试

1. **安装 Flipper**
   - 下载地址：https://fbflipper.com/

2. **安装插件**
   - 安装 Networking 插件
   - 可以实时查看、拦截、修改网络请求

3. **模拟不同场景**
   - 查看请求和响应
   - 模拟网络超时
   - 修改响应数据

## 🎯 完整验证流程

### 场景一：完整用户旅程

1. **在线模式，一切正常**
   - ✅ 成员列表加载正常
   - ✅ Toast 显示成功消息（如果有）

2. **网络不稳定**
   - 切换到「网络限流」模式，失败率 50%

3. **完全离线**
   - 切换到「离线」模式
   - ✅ 顶部显示「离线模式」横幅
   - ✅ Toast 显示网络错误
   - ✅ 操作被添加到离线队列

4. **网络恢复**
   - 切换回「正常网络」模式
   - ✅ 离线横幅消失或变为待同步提示
   - ✅ 队列中的操作自动同步

## 🎓 最佳实践建议

### 1. 错误处理

- ❌ 不要：每个 API 调用都写 try/catch 重复代码
- ✅ 应该：使用封装好的 API 请求层

### 2. 防重复点击

- ❌ 不要：依赖禁用按钮状态（容易被绕过）
- ✅ 应该：使用 `usePreventDoubleClick` Hook

### 3. 重试策略

- ❌ 不要：所有错误都无限重试
- ✅ 应该：只对网络错误等可重试错误进行指数退避重试

## ❓ 常见问题

### Q: 为什么有些错误不会自动重试？

A: 只有可重试的错误（网络错误、超时、500 错误）才会自动重试。客户端错误（403、404、422）不会重试，因为重试也不会成功。

### Q: 如何自定义某个 API 的重试次数？

A: 在调用 `executeApiRequest` 时，通过 `maxRetries` 选项覆盖默认值：

```typescript
executeApiRequest({
  id: 'my-request',
  requestFn: myApiCall,
  maxRetries: 5, // 自定义重试次数
});
```

### Q: 防重复点击的延迟时间可以调整吗？

A: 可以，`usePreventDoubleClick` Hook 接受 `delay` 参数：

```typescript
const { isLoading, executeWithLock } = usePreventDoubleClick(1000);
```
